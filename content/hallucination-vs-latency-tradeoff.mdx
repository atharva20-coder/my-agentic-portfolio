## The Core Tension

When you deploy a voice AI agent — especially one that makes phone calls on behalf of a business — two failure modes will ruin you:

**Failure mode A:** The bot says something wrong. It quotes an incorrect loan amount. It confirms a payment date that doesn't exist. It promises a policy that isn't offered. The customer is now misinformed, possibly legally.

**Failure mode B:** The bot pauses too long. Every response has a 2-second gap. The customer asks "hello? are you there?" three times and hangs up.

Here's the uncomfortable truth: **the techniques that reduce mode A make mode B worse, and vice versa.**

This is the hallucination-latency tradeoff. It's not a bug you can fix. It's an architectural constraint you have to manage.

## Why Accuracy Fixes Add Latency

### Retrieval-Augmented Generation (RAG)

RAG is the standard solution for grounding LLMs in factual information. Instead of relying on the model's parametric memory, you retrieve relevant documents at query time and inject them into the context.

In a voice agent context:
1. User's utterance arrives
2. Embed the utterance
3. Vector search against your knowledge base
4. Retrieve top K chunks
5. Inject into LLM context
6. Generate response

Steps 2–4 take time. Even a fast retrieval system adds 100–300ms per turn. In a voice call where you're targeting 600ms total TTFT, that's 17–50% of your entire budget.

### Verification Steps

For high-stakes facts — loan amounts, EMI dates, product terms — you might run a separate verification pass. The LLM generates a response, a second call checks it against a database, then you output the response.

This serializes two LLM calls. You're now at 2x the base latency minimum.

### Longer System Prompts

More constraints in the system prompt generally reduce hallucination. "Never quote interest rates not in the provided data. Never confirm payment receipt without a confirmation ID." Every additional instruction anchors the model better.

But more context = more tokens = slower first token generation. A 5KB system prompt is noticeably slower than a 500-byte one.

### Thinking/Reasoning Mode

Models with explicit reasoning — chain-of-thought, `thinkingLevel=high` in Gemini — hallucinate less. They're more likely to catch internal contradictions before outputting them.

`thinkingLevel=high` vs `thinkingLevel=minimal` in my testing on Gemini Live: roughly 200–400ms additional TTFT per turn, but measurably lower errors on edge cases.

## Why Latency Fixes Add Hallucination Risk

### Minimal Thinking

Set `thinkingLevel=minimal` and you get the fastest possible first token. The model doesn't reason through the prompt — it pattern-matches and outputs.

Pattern matching is fast. It's also where hallucinations live.

```python
thinking_config=types.ThinkingConfig(thinking_level="minimal")  # fastest, most hallucination-prone
```

vs.

```python
thinking_config=types.ThinkingConfig(thinking_level="low")  # 200ms slower, noticeably more accurate
```

### Smaller Context Windows

Truncating context to reduce prompt size keeps TTFT low but means the model has less grounding information. When a customer asks about a specific offer from a previous call, a truncated context might not have it — and the model will fill the gap with something plausible but wrong.

### Smaller Models

Smaller models are faster. Flash is faster than Pro. Haiku is faster than Opus. But smaller models hallucinate more, especially on domain-specific or numerically precise content.

In BFSI voice agents, this is critical. A small model confidently stating the wrong EMI amount on a recorded call is a compliance incident.

### No RAG

Skip retrieval and rely on the system prompt alone. Fast, but the model is working from memory — and LLM memory on specific product terms, loan details, and regulatory constraints is unreliable.

## How to Think About the Tradeoff

The tradeoff isn't binary. It's a dial, and different parts of your conversation can be dialed differently.

### Not all turns have equal stakes

A turn where the user says "okay, I'll pay by Friday" is low-stakes. Any reasonable response works.

A turn where the user says "what's my current outstanding balance?" is high-stakes. Getting that wrong is a legal problem.

You don't need to treat every turn the same. Route high-stakes turns through verification. Let low-stakes turns run fast.

```python
HIGH_STAKES_INTENTS = {
    "request_balance",
    "confirm_payment_amount",
    "request_interest_rate",
    "identity_verification",
}

async def generate_response(intent: str, context: dict):
    if intent in HIGH_STAKES_INTENTS:
        # Slower path: retrieve from database, inject fact, verify
        facts = await fetch_account_facts(context["account_id"])
        return await generate_with_grounding(facts, context)
    else:
        # Fast path: direct generation from system prompt
        return await generate_fast(context)
```

### Pre-compute what you can

If you know the account balance, EMI date, and loan number before the call starts, inject them at session creation. They're not retrieved on demand — they're already in context.

The retrieval cost is paid once, at session start, outside the TTFT window. The model always has the facts. No mid-turn retrieval latency.

```python
# Pre-fetch account data at call start
account_data = await fetch_account_details(customer_id)

system_prompt = f"""
You are Aria, a collections agent.

CUSTOMER ACCOUNT FACTS:
- Outstanding balance: ₹{account_data['outstanding']}
- EMI date: {account_data['emi_date']}
- Last payment: {account_data['last_payment_date']}
- Loan ID: {account_data['loan_id']}

Do not quote any amounts or dates not listed above.
"""
```

Zero per-turn retrieval latency. Always grounded on the critical facts.

### Constrain the model's output space

One of the most effective hallucination reduction techniques is limiting what the model is *allowed* to say. Instead of asking the model to generate free-form responses, constrain it to a decision tree.

For a collections call, the decisions are finite: acknowledge, probe, handle objection, offer arrangement, escalate, close. Each stage has a narrow response space.

A model that's choosing between 5 things hallucinates less than a model generating arbitrary text. And it's faster, because the output is shorter.

### Use `thinkingLevel=low`, not `minimal`

After extensive testing with Aria, `minimal` thinking had a meaningful hallucination rate on edge cases. Switching to `low` reduced that substantially with ~200ms additional latency.

For a conversation that's already well-structured and pre-grounded, `low` is the right tradeoff. `minimal` is appropriate only for turns where no factual claim is being made.

## The Measurement Problem

Here's what makes this tradeoff genuinely hard: **you can't measure hallucination rate in real time.**

Latency you can measure per turn, per call, continuously. Hallucination requires human review, or a separate evaluation LLM, or post-hoc analysis of call recordings and CRM data.

This means you often don't know you have a hallucination problem until it's caused an incident. A customer was told the wrong EMI date. A dispute was filed. A call recording was pulled.

Build monitoring that catches these retrospectively:

1. Log every factual claim the model makes (amounts, dates, IDs)
2. Compare against CRM truth data after each call
3. Flag discrepancies for human review
4. Track discrepancy rate by intent, stage, and model config

This won't prevent hallucinations. But it'll tell you exactly where they're happening so you can address them systematically.

## The Honest Bottom Line

There's no configuration where you get zero hallucination and zero latency penalty. The tradeoff is real.

The way to navigate it:

1. **Pre-ground critical facts** at session start, outside the TTFT window
2. **Route high-stakes turns** through verification; let low-stakes turns run fast
3. **Constrain the output space** wherever possible — fewer choices, fewer errors
4. **Use `thinkingLevel=low`** not `minimal` for any call where factual accuracy matters
5. **Measure hallucination retrospectively** against ground truth, not just subjectively
6. **Accept different latency budgets** for different turn types — not every turn needs to be 600ms

The goal isn't to eliminate the tradeoff. It's to make smart decisions about where in the conversation you're willing to be slower, and where you can afford to be fast.

In a collections call, the introduction can be fast. The balance confirmation cannot. The closing can be fast. The arrangement terms cannot.

That differentiation — not a universal setting — is what separates a voice agent that performs from one that either sounds wrong or sounds robotic.
